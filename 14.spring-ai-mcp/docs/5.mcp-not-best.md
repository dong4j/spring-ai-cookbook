大纲示例：
1.	MCP 很好，但不是银弹
  - 标准化了：协议、工具描述、调用流程
  - 没有标准化：安全边界、权限模型、调用策略、上下文管理
  - 从「多爽」回到「多难」的现实
2.	安全层面的隐患
  - MCP 工具常常意味着强权限：文件系统、shell、内部服务
  - 模型 + 工具 = 远程 root
  - Prompt 注入风险：用户 / 文本可以诱导模型调用危险工具
  - 缺少：权限边界、人工审批、策略引擎
3.	上下文层面的隐患
  - MCP 工具 / 资源都需要 schema / 描述
  - 工具一多，描述一堆，模型上下文吃不消
  - 工具发现靠「全量注入」会导致：
    - token 花费增加
    - 模型选择困难（工具选择 becomes non-trivial）
4.	治理与可控性问题
  - 谁能控制哪些工具在哪些场景可用？
  - 谁能审计工具调用日志？
  - MCP 协议本身不关心这些，交给上层应用
5.	小结：MCP 解决了“工具调用标准化”，但留下了“工具治理”和“工具规划”问题
  - MCP = 协议层
  - 工具治理 & 调度 = 还没标准
  - 下一篇我们就通过一个具体的设计（Claude Skills & 你自己实现的 skill runtime）来看：在 MCP 之上，如何再加一层「任务级抽象」来管理这些问题。

过渡到第 7 篇：

MCP 把「工具」标准化了，但没有把「任务」标准化。
所以业界开始在 MCP 上叠加一层“技能（Skill）”抽象：
- 模型不再直接面对一堆 MCP 工具
- 而是只面对少量“任务级技能”
下一篇，我们就正式进入这个话题，并用 Spring AI 实现一个简化版的 Skill 系统。
